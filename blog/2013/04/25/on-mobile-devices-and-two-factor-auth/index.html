A Note on the Security of Mobile Banking Apps

Apr 25th, 2013

My bank, MITFCU, just came out with a mobile banking app. Like similar apps from other financial institutions, the app allows you to view your accounts, deposit checks, transfer money, and so on. It’s both useful and convenient, but it has a couple of potentially serious security issues.

Insecure two-factor authentication

Like any security-conscious person, I had two-factor auth turned on for my bank account. Two-factor authentication in its basic form relies on the combination of “something you know” (generally a password) with “something you have” (a phone number for SMS, email address, RSA fob, smart card, etc.)

Whenever I log in to MITFCU’s desktop site, I receive a text message with a six-digit code, which I must provide in order to continue logging in. All well and good.

When I first logged into the mobile banking app, a popup appeared saying, approximately, “We’re sending you a code via SMS. Please type it in.” I immediately received the SMS on the same phone. Can you see the problem here?

If an adversary has my phone, they’ve already got the “something I have.” My two-factor auth security has been reduced to a single factor: my password.

There are a few cases where it makes sense, like if the attacker’s installing the app on a different phone.

Ultimately, however, when using the app on the same phone you use to log in, you’ve reduced your two-factor auth to a single factor.

I’d rather see a system where you have to use a previously-authenticated computer to get the first-time login code – think the reverse of the traditional two-factor auth approach.

Password complexity and the tyranny of the soft keyboard

Then there’s the other credential: the password. MITFCU’s app logs you out after a short time, whereupon you must reenter your login password.
